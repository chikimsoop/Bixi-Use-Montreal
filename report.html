This report highlights the development of the BIXI network in Montreal, seeking to identify the key successes in its design. Hopefully, other regions can learn from this project to develop a strategy to implement their own system.

To highlight BIXI's expansion strategy, we decided to create a map showcasing how the network has changed from year to year. We included data from 2014 to 2024, as that was the data publicly available (BIXI, 2025b).

To help us plan our approach, we brainstormed then illustrated a storyboard


Figure 1. Storyboard created by the group, illustrated by Kimsoo Gopnik

To start, we show the types of bikes available through the program, to give the audience more of an idea of the use of the network. We found stock images of a classic and an electric BIXI, and ensured the photo files were uploaded to cyberduck and not on a local drive. (BIXI Montreal, 2022). The slider was made using the "JuxtaposeJS" feature on Northwestern University's "knight lab" website. We downloaded images and aligned them using Canva, then uploaded them to the Juxtapose site, which provided code to embed them.

The next chapter includes a video introducing BIXI, which we embedded using a technique learned on W3Schools. This video is meant to give the audience a basic idea of the concept, to contextualize the following sections (BIXI Montreal, 2016). To create the iframes for the BIXI type slider and the embedded Youtube video, we created an element in the html file that could be called on in the javascript. If the record has an attribute 'iframe', it will then create an iframe with 'source' link given in javascript. To embed the Youtube video, Youtube provides embedded code as an option when sharing the video.

For the first geographic chapter, we examine the temporal expansion of the network. This series of maps was inspired by the agency's own yearly announcements of the network's expansion (BIXI, 2025a), seeking to visualize it in an accessible format. Every year, new stations for that year are marked in green, which we figured out how to code using the Mapbox website (Mapbox). We used a 20 meter buffer (in ArcGIS Pro) to determine if a station was new to try to avoid small movements of stations year to year. However, this was not perfect, as a few stations moved as much as a few blocks. Nevertheless, the expansion and densification of the network year by year is clearly visible, demonstrating the BIXI and its municipal partners' desire to make the service available to more and more users and increase its utility, compounding its success.

The point function was built to control how stations appeared and changed year by year. The main styling function accepted a year as input, then used conditional expressions (case statements) to compare each station's (Year_Est) attribute with the selected year. If the station opened that year, it was colored green; if it already existed, it was colored blue. Opacity was reduced if a station's (Year_Est) was greater than the current year, or if the station had a removal year later than the current year. Since Mapbox does not allow variables to be passed directly between functions, we created helper functions for each year that called the main function with the correct input (an approach suggested by a kind stranger in the GIC who helped us troubleshoot the issue). This allowed us to automate the visualization of yearly changes, so that as users scrolled through the storymap, the points dynamically updated to highlight only the stations relevant to that year.

To visualize the BIXI station distribution in a different way, we included a choropleth map of the number of stations by borough and municipality for 2024 (Ville de Montreal, 2025a). The number of stations per municipality was determined by taking the 2024 BIXI stations and performing a spatial join in ArcGIS Pro. In Mapbox this was visualized by setting the color value to a conditional statement. Then, we visualized the current bike lane network for Montreal (Ville de Montreal, 2025b), to illuminate the extent of the infrastructure necessary to enable the widespread adoption of the BIXI system. The bike lanes are distinguished as protected or not protected via conditional statements in the chapter.

To show a more granular example of the everyday utility of the BIXI system, we zoomed in on a three-dimensional model of downtown Montreal. To create this map, we subtracted the average given elevations of the buildings from the elevation of the ground, to find the actual heights of the buildings (Ville de Montreal, 2016; Natural Resources Canada, 2025).

Finally, a potential trajectory from the Sherbrooke Metro station (logo: Societe de transport de Montreal, 2010) to the McGill campus (logo: Wikipedia, 2025) is shown, highlighting the seamless convenience of BIXI as a commute mode.

Appendix 2: Individual reflection - Nick Morton

Parts responsible for
I downloaded the administrative boundaries from the Montreal Open Data Portal and converted them into a GeoJSON in ArcGIS Pro. I created the code for adding points, polygons, lines, and buildings to the map and customized their styles. I also built the function to place the BIXI stations on the map by year, including writing separate functions for each year to pass the year into the main styling function. In addition, I developed the walking route from the metro station to McGill by creating points and lines in ArcGIS Pro and linking them through multiple chapters in the storymap. I uploaded the McGill and STM logos, helped with overall styling, cleaned up the code, and adjusted opacity so that layers displayed properly when users scrolled back up.

Customization techniques used
A key technique I used was applying conditional expressions (case) to filter and display specific attributes within Mapbox layers. For example, I isolated protected bike lanes from the full dataset, and I used conditional styling to adjust color and opacity for lines, polygons (to show how many bike lanes were in each area), and points (through the year function as well as selecting two specific points for the bike route). Learning how to use (case) was the most important customization technique. To represent temporal change, I created functions that changed the color and opacity of BIXI station points by year, calling the main function through a year-specific function. I also created (if) statements in the style part of the html that allowed me to change the color for all data types. Finally, I designed the Bike to McGill chapter, which chained together chapters with changes in location and heading, making it feel like a guided route through the city.
To break it down by technique: the points were best for showing how the BIXI stations changed year by year, which made the expansion immediately clear. The polygons worked best for showing counts, like the number of stations per borough or municipality, so we could compare areas. The lines were important for connectivity, since they showed where protected versus unprotected bike lanes were. The 3D buildings didn't change, but they showed the city in context. Finally, the bike chapter was useful for showing how BIXI's fits into peoples everyday life.

Hoped to do compared to actually accomplished
We successfully built the features outlined in our original storyboard, over time our ambitions grew and we turned out to be overly ambitious for the time available. For example, we planned to expand the project with more advanced interactivity. I was hoping to add an animated line as seen in this mapbox tutorial

Use of dynamic and spatial data to improve the narrative process
Static spatial data, such as the administrative boundaries and 3D buildings, provided consistent geographic context in the story. Dynamic spatial data, particularly the annual changes in BIXI station locations, demonstrated the rapid growth of the system. The walking route from the metro station to McGill added a more personal narrative, showing users how BIXI bikes can realistically fit into moving through the city.
